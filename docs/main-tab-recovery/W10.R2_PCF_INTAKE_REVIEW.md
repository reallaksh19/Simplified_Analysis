# W10.R2 — PCF Intake and Review Recovery

## Runtime boundary

W10.R2 implements the existing `PCF` application identity as a framework-neutral current-runtime consumer. The shipped entrypoint remains `src/main.js`, the application continues to use one `ApplicationShellController`, and `WorkspaceState` remains the only accepted-dataset authority.

The increment introduces closed successors:

```text
workspace-consumer-registry/v6
application-view-state/v6
pcf-intake-source/v1
pcf-review-model/v1
pcf-review-export/v1
```

Registry and view-state versions v1 through v5 remain independently constructible. Navigation identity and order remain unchanged from W10.R1.

## Transactional intake

The PCF consumer uses this boundary:

```text
selected file or typed PCF text
→ staged controller text
→ explicit parse
→ immutable intake source
→ immutable read-only review
→ explicit Adopt into Workspace
→ existing dataset:loadRequested event
→ existing DatasetController and WorkspaceState
```

Selecting a file, typing, opening the tab, parsing, reviewing, cancelling, or exporting does not replace the active Workspace dataset. Only the explicit adoption action publishes the existing dataset-load request.

## Source and review evidence

`pcf-intake-source/v1` retains normalized source text and identity, declared units, deterministic component identities, source-authored order, source line ranges and raw lines, endpoint/centre/branch evidence, source attributes, diagnostics, canonical-geometry preview evidence, and an explicit adoption decision.

The semantic intake identity is independent of component encounter order when records are semantically equivalent. Source text identity and source-order evidence remain separate and visible.

`pcf-review-model/v1` projects the complete staged review without introducing edit authority. Source-derived values are written to DOM text nodes or form values, never interpreted as HTML.

## Adoption blocking

Adoption fails closed for empty or unparsable source; missing, unsupported, duplicated or inconsistent units; invalid coordinates; duplicate component identities; missing required geometry; failed canonical projection; an empty segment set; or staged text changed after parsing.

Review-only component types remain visible with explicit diagnostics and are excluded from the Workspace package rather than silently converted.

## Workspace package

Successful adoption creates an `inputxml-managed-stage/v1` package with deterministic source IDs, millimetre geometry, retained PCF attributes and source-line ancestry. The package crosses the existing `dataset:loadRequested` boundary and is normalized by the existing `DatasetController`; W10.R2 does not directly mutate `WorkspaceState`.

## Export

The PCF consumer provides deterministic JSON and CSV review exports. Export contains review evidence only and does not adopt the source or modify Workspace state.

## Excluded scope

W10.R2 does not provide component or attribute editing, PCF writeback, automatic repair, automatic fitting construction, Sketcher launch, solver execution, FEA, stress calculations, a second viewport, React/Zustand restoration, Local Stress integration, or Element FEA integration.
