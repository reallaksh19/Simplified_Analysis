# W10.4 Reusable Load Cases and Model Load Primitives

## Scope

Mission W10.4 builds deterministic component-level mass and gravity-load primitives from `shared-piping-model/v1` and `piping-port-topology-graph/v1` only.

It does not add routes, chainage, support allocation, reactions, equilibrium, load combinations, solver execution, global force vectors, viewport behavior, authoring, or reporting changes.

## Contracts

- `gravity-profile/v1`
- `load-case-set/v1`
- `engineering-load-source-projection/v1`
- `load-composition-profile/v1`
- `model-load-primitive-set/v1`
- `model-load-readiness-audit/v1`

All contracts are deeply immutable and use deterministic semantic hashes.

## Gravity and quantity separation

`STANDARD_GRAVITY_9_80665_V1` exposes `9.80665 m/s²` and semantic direction `GRAVITY_DOWN`. Gravity is not hidden inside formulas.

Mass remains in kg or kg/m. Weight force remains in N or N/m. Gravity primitives carry `globalVector: null` because no certified load frame exists.

## Load cases

EMPTY, OPE and HYD are extensible records rather than permanent case-specific fields.

- EMPTY includes pipe metal, insulation and component mass.
- OPE adds operating fluid.
- HYD adds hydrotest fluid.

Missing operating-fluid evidence blocks only OPE. Missing hydrotest evidence blocks only HYD.

## Composition and formulas

`PIPING_COMPONENT_LOAD_COMPOSITION_V1` exposes the linear/lumped type classification and direct-source precedence.

Versioned formulas:

- `PIPE_METAL_MASS_PER_LENGTH_V1`
- `FLUID_MASS_PER_LENGTH_V1`
- `INSULATION_MASS_PER_LENGTH_V1`
- `MASS_TO_WEIGHT_FORCE_V1`

Each derived result retains canonical substitutions, source inputs and paths, raw output, canonical unit, rounding rule and `fallbackUsed: false`.

Direct zeroes are valid. Negative inputs are rejected. Missing geometry, mass, COG, dimensions, density, insulation or fluid evidence remains blocked rather than defaulted.

## Double-count prevention

Linear pipes may emit distributed mass. Lumped fittings use explicit component mass at an explicit COG or center. A component carrying conflicting distributed and lumped mass evidence is blocked with `DOUBLE_COUNT_CONFLICT`.

No midpoint is invented for a point load. A point moment is emitted only from explicit moment and axis evidence at a qualified explicit COG or center.

## Workspace capability

The right panel displays independent EMPTY/OPE/HYD readiness, primitive counts, mass totals, force totals and active profiles.

Actions:

- `Rebuild Model Loads`
- `Export Load Case/Primitive Model`

Model preparation may occur after shared-model/topology changes. It does not publish analysis or solver events. Export is explicit, deterministic, dataset-derived and object-URL-safe.

Read-only getters expose the current load-case set, primitive set and readiness audit.

## Known limitations

- Component loads are limited to evidence already preserved by the shared model.
- Unknown types are blocked unless qualified direct source mass is available.
- No route, support, reaction or solver readiness is claimed.
- No permanent moment is derived from guessed actuator offsets.
- No global load vector is emitted.
