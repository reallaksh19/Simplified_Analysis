# W10.1 Shared Piping Model Foundation

## Scope

Mission W10.1 introduces a framework-neutral engineering evidence boundary between the current Analysis Workspace dataset and legacy canonical geometry consumers.

The mission adds three contracts:

- `source-package-snapshot/v1`
- `staged-model-index/v2`
- `shared-piping-model/v1`

It does not add topology, inferred connectivity, route chainage, support attachment, restraint interpretation, load cases, support reactions, solver inputs, or solver results.

## SourcePackageSnapshot.v1

The source snapshot preserves the parsed JSON package and records two distinct hash concepts:

- `sourceSemanticHash` hashes canonical JSON content and is stable across object-key ordering.
- `sourceByteHash` hashes original upload bytes only when those bytes are explicitly supplied.

The current Analysis Workspace receives an already parsed object at the dataset adapter boundary. Therefore `sourceByteHash` is `null`, accompanied by `SOURCE_BYTES_UNAVAILABLE`. This is not a claim that canonical JSON is byte-identical to the uploaded file.

## StagedModelIndex.v2

The staged index stores one lightweight record per source-tree occurrence. Each record contains:

```text
sourceNodeKey
sourceEntityId
jsonPointer
parentSourceNodeKey
childSourceNodeKeys
childIndex
depth
type
name
sourcePath
lineId
branchId
systemId
zoneId
diagnostics
```

`sourceNodeKey` is based on the source JSON pointer and is unique even when source entity IDs are duplicated or missing. Duplicate and missing source IDs remain visible in the validation audit and source snapshot.

The index includes deterministic lookup records by node key, source entity ID, line ID, branch ID, type, and source path. Full source attributes, enrichment payloads, and native parameters remain in the single source snapshot rather than being copied into every index row.

## SharedPipingModel.v1

The shared model contains reusable engineering evidence only:

- project and explicit unit evidence;
- source snapshot reference;
- components and supports;
- unconnected component ports;
- line, branch, system, and zone identity evidence;
- normalized geometry evidence;
- normalized engineering property evidence with source paths;
- source references and deterministic diagnostics;
- summary and indexes.

Ports are explicitly unconnected. Their presence does not imply connectivity or topology.

## Compatibility adapters

Four named-export pure adapters are provided:

```text
analysis-workspace-dataset/v1 -> shared-piping-model/v1
canonical-geometry-v1 -> shared-piping-model/v1
shared-piping-model/v1 -> canonical-geometry-v1
shared-piping-model/v1 -> calculation-workspace/v1
```

The calculation Workspace bridge consumes the shared model projection directly. It does not invoke the raw staged-package normalizer or traverse `sourceModel.sourcePackage` again.

Compatibility projections preserve supported fields and emit deterministic diagnostics for unsupported or structurally lossy fields. No value is invented to make a projection pass.

## Workspace export

The Analysis Workspace right panel displays shared-model component, support, port, and diagnostic counts. `Export Shared Model` is an explicit user action that:

1. validates the shared model;
2. serializes canonical UTF-8 JSON deterministically;
3. uses a dataset-derived filename;
4. dispatches a browser download;
5. revokes the object URL;
6. leaves Workspace state unchanged.

`globalThis.AnalysisWorkspace.getSharedModel()` returns the current deeply frozen model or `null` when the Workspace is empty.

## Known limitations

- Original upload bytes are unavailable at the current Workspace adapter boundary, so `sourceByteHash` remains `null`.
- Legacy canonical loads and material-library collections are outside this contract and produce diagnostics.
- Canonical record fields outside the supported identity, geometry, dimensional, material-name, and reference fields remain source-preserved but diagnostic in compatibility conversion.
- No component ports are connected.
- No support semantics are inferred.
- No calculation runs automatically.
