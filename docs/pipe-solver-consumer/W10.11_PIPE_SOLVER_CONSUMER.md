# W10.11 — Guarded Pipe Solver Consumer Adoption

## Scope

W10.11 adopts the existing `pipe-screening` capability as a framework-neutral `PIPE_SOLVER` consumer. It does not add or modify a solver, formula, engineering default, unit conversion, session owner, ledger owner, coordinator, exporter, viewport or dormant application interface.

```text
workspace-consumer-context/v1
+ exact selected Workspace entity
+ exact pipe-screening inspection/readiness
+ exact matching analysis-session/v1 | null
+ exact current-dataset pipe-screening ledger evidence
→ pipe-solver-consumer-source/v1
→ pipe-solver-review-model/v1
→ PIPE_SOLVER view
→ existing validated analysis events only
```

## Authority boundaries

The existing owners remain authoritative:

- `WorkspaceState` owns the dataset and selected entity.
- `AnalysisCapabilityRegistry` and `pipeScreeningCapability` own applicability, input inspection, assumptions, limitations and readiness.
- `AnalysisSessionController` and `AnalysisSessionStore` own session identity, overrides, field errors and lifecycle.
- `AnalysisCoordinator` and the existing capability own execution.
- `solver-result-contract-v1` owns accepted result structure.
- `AnalysisLedgerStore` owns sequence, active entry, retention and comparison state.
- `AnalysisLedgerController` and the existing report exporter own export scope.

The new Workspace adapter receives the exact existing registry and store instances through explicit injection. The core package receives no runtime owner and performs no mutation.

## Source contract

`pipe-solver-consumer-source/v1` retains:

- current dataset, workspace and consumer-context identity;
- a contained selected-entity projection with exact source-attribute and native-parameter references;
- exact capability input/readiness evidence;
- the exact active session only when capability, dataset, target and current Workspace identity match;
- exact matching ledger-entry references ordered by sequence and then entry ID;
- the authoritative active ledger ID only when it belongs to the retained set;
- canonical diagnostics for every exclusion or blocked condition.

No missing value is filled. Explicit zero is preserved. Diagnostic arrival order does not affect semantic identity.

## Review model

`pipe-solver-review-model/v1` consumes only a validated source snapshot. Public validation reconstructs the expected model and checks exact source-reference identity.

It projects:

- selection and capability summaries;
- exact input values, units, provenance paths, editability, committed overrides and field errors;
- matching session lifecycle evidence;
- an accepted current result only when schema, module, engineering level, geometry method and semantic integrity evidence are valid;
- matching ledger history without combining or choosing a governing result;
- exact assumptions, limitations, failures, warnings, diagnostics and formula traces.

## Method identity separation

The existing runtime has two valid method identities with different scopes:

```text
Capability manifest / workflow family:
SIMPLIFIED_2D_TOPOLOGY_SCREENING

Exact solver-result method:
SIMPLIFIED_2D_<GEOMETRY_TYPE>
```

The consumer preserves both. It never rewrites the exact solver result to match the manifest.

Accepted result evidence must have:

```text
schemaVersion = solver-result-contract-v1
moduleId = simplified-2d
engineeringLevel = BENCHMARKED_SCREENING
methodId in the closed geometry-method set
meta.geometryType exactly linked to methodId
```

Closed linkage:

```text
EMPTY             → SIMPLIFIED_2D_EMPTY
L_SHAPE           → SIMPLIFIED_2D_L_SHAPE
Z_SHAPE           → SIMPLIFIED_2D_Z_SHAPE
LOOP_OR_MULTI_LEG → SIMPLIFIED_2D_LOOP_OR_MULTI_LEG
OFFSET            → SIMPLIFIED_2D_OFFSET
SINGLE_LEG        → SIMPLIFIED_2D_SINGLE_LEG
STRAIGHT_RUN      → SIMPLIFIED_2D_STRAIGHT_RUN
MULTI_LEG fallback → SIMPLIFIED_2D_EMPTY
```

Unknown methods, unknown geometry types and mismatched method/geometry pairs are excluded and diagnosed. The capability manifest remains `SIMPLIFIED_2D_TOPOLOGY_SCREENING`.

## Readiness separation

Generic `PIPE_SOLVER` navigation availability requires only:

```text
sharedModel
topologyGraph
topologyAudit
```

Inside the view, separate evidence is shown for:

```text
selection availability
capability applicability
readyToReview
readyToRun
active-session state
field errors
individual action eligibility
```

Generic availability is not made selection-dependent.

## Existing event boundary

The view may publish only:

```text
analysis:sessionOpenRequested
analysis:sessionOverrideRequested
analysis:sessionResetRequested
analysis:sessionCloseRequested
analysis:requested
analysis:ledgerActiveRequested
analysis:exportRequested
```

Every action is reevaluated against the current validated review model immediately before publication. The existing event validator and runtime owners remain the final mutation and execution boundary.

## Export semantics

The export control is labeled as export of the current analysis report. It does not claim to export only a selected Pipe Solver row because the existing export event accepts only a format and uses authoritative ledger active/comparison state.

## Engineering claim containment

The only authorized policy is:

```text
EXISTING_BENCHMARKED_SIMPLIFIED_2D_SCREENING_ONLY
```

Prominent limitations are retained:

- Benchmarked simplified 2D screening only.
- Not final piping-code stress analysis.
- Not general 3D piping flexibility.
- No branch flexibility, equipment interaction, nonlinear gaps, friction or restraint stiffness.
- No automatic execution.
- Inputs must be explicit or user-overridden through the existing session boundary.

## Compatibility

W10.11 adds:

```text
workspace-consumer-registry/v4
application-view-state/v4
```

Factories and validators for v1, v2 and v3 remain closed and unchanged. Existing public view-state behavior is retained for Workspace, Reports, Load Calc and 3D Calc; the public API exposes v4 while Pipe Solver is active.
